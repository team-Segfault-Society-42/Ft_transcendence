# Security Policy

## Project Context

This repository is maintained by a team of five students as part of the 42 school project **ft_transcendence**.

Because this is a study project, security support is handled on a best-effort basis by the team.

## Reporting a Vulnerability

If you discover a security issue, please **do not open a public issue**.

Instead, report it privately to the team through the usual private communication channel.

## What to Report

Examples of security-related issues include:

- leaked `.env` values
- exposed API keys, tokens, or passwords
- authentication or authorization flaws
- insecure Docker, Nginx, or backend configuration
- dependency vulnerabilities with a real impact on the project
- accidental exposure of private or sensitive data

## What to Include in a Report

When reporting a security issue, please include:

- a clear description of the problem
- the part of the project affected
- steps to reproduce, if possible
- the possible impact
- any suggested fix, if you have one

## Responsible Handling

Please avoid sharing the issue publicly before the team has had time to review and fix it.

This helps protect the project, the repository, and any sensitive configuration that may be involved.

## Scope

This policy applies to the whole project, including:

- frontend
- backend
- database
- Docker configuration
- reverse proxy configuration
- environment and secret management

## Team Response

The team will review reported issues, discuss the appropriate fix, and handle the update according to the seriousness of the problem and the project timeline.
